# 卡片主线与流水线布局思考

## 一、论文到底怎么发？——科研的几种「范式」

你提到的 A+B 模式，确实是 CS 领域最常见的论文范式，但不是唯一的。我梳理了一下，大概有这么几种：

### 范式 1：方法迁移（A+B）—— 最常见

> 从别的领域搬一个方法过来，用在自己的问题上。

流程：读已有工作 → 发现问题的瓶颈 → 从其他领域找到一个方法 → 适配过来 → 跑实验 → 写论文

这是最"安全"的路径，也是大多数硕士生的主路径。你说的"距离比较远的论文的方法能不能抄过来"，就是这个范式的核心直觉。

### 范式 2：新问题定义 —— 较少见但影响力大

> 不是提出新方法，而是提出一个没人问过的问题。

例子：Attention Is All You Need 不是改进了某个旧方法，而是说"我们根本不需要 RNN"。BERT 不是改进了某个分类器，而是提出了"预训练+微调"这个新范式。

流程：大量阅读 → 发现现有工作都在同一个框架里打转 → 提出一个不同的问题定义 → 用（可能很简单的）方法验证这个问题值得研究

### 范式 3：系统/工具论文 —— 实用性强

> 不是提出新方法，而是造了一个好用的工具/系统。

例子：Hugging Face Transformers、OpenAI Gym、COCO 数据集。

流程：发现社区缺一个工具 → 设计并实现 → 让别人用起来 → 写论文描述设计决策和使用情况

### 范式 4：理论分析 —— 门槛高

> 不做新实验，而是从数学上解释"为什么现有方法有效"或"在什么条件下会失效"。

流程：观察到一个经验现象 → 建立数学模型 → 推导证明 → 用实验验证理论预测

### 范式 5：实证研究/Benchmark —— 基础但重要

> 不提新方法，而是做一次大规模的公平对比，或者揭露某个被忽视的问题。

流程：发现不同论文的实验设置不一致 → 统一设置重新对比 → 发现某些"结论"其实不成立 → 或者发现新的规律

### 范式 6：负面结果/反思论文 —— 少见但有价值

> 证明某个被广泛使用的方法其实没用，或者某个被忽视的因素其实很重要。

---

## 二、你的直觉是对的：A+B 确实是主线

虽然有上面 6 种范式，但在实际操作中，**范式 1（方法迁移）占据了绝大多数**。原因很简单：

- 新问题定义需要极深的领域洞察，大多数研究生做不到
- 系统论文需要大量工程投入，周期长
- 理论分析需要很强的数学功底
- 实证研究和负面结果在顶会中接受率偏低

所以对于你的卡片系统来说，**主线就是范式 1 的流程**。其他范式可以作为分支或变体。

## 三、主线流程拆解：从「读论文」到「发论文」

我试着把范式 1 的完整流程拆成一条线性的主线，每一步的输出就是下一步的输入：

```
┌─────────────────────────────────────────────────────────┐
│  主线：一篇论文从无到有的完整路径                           │
└─────────────────────────────────────────────────────────┘

Step 1: 定方向
  输入：你的研究兴趣/关键词
  输出：检索策略 + 初步论文列表
  卡片：文献检索策略

Step 2: 读论文（批量）
  输入：论文列表（来自 Step 1）
  输出：每篇论文的结构化笔记
  卡片：论文精读笔记（反复使用）

Step 3: 建立全局观
  输入：多篇精读笔记（来自 Step 2）
  输出：领域图谱（演进脉络 + 流派 + 空白）
  卡片：领域图谱梳理

Step 4: 找到「迁移源」
  输入：领域图谱中的空白和瓶颈（来自 Step 3）
  输出：发现某个「别的领域的方法」可能解决我们领域的问题
  这一步目前没有对应卡片！

Step 5: 想 Idea
  输入：领域空白 + 迁移源方法 + 自己的初步想法
  输出：5 个候选 Idea
  卡片：研究 Idea 头脑风暴

Step 6: 筛 Idea
  输入：候选 Idea（来自 Step 5）
  输出：可行性评估 + 创新点提炼
  卡片：可行性评估 + 创新点提炼

Step 7: 设计方法
  输入：确定的 Idea + 创新点（来自 Step 6）
  输出：完整技术方案（架构 + 模块 + 公式）
  卡片：方法架构设计

Step 7.5: 设计训练策略
  输入：方法架构（来自 Step 7）
  输出：损失函数 + 优化策略
  卡片：损失函数与优化策略

Step 8: 设计实验
  输入：方法架构 + 核心贡献（来自 Step 7）
  输出：实验方案 + 消融方案
  卡片：实验方案设计 + 消融实验设计

Step 9: 跑实验
  （这一步是手动的，不是卡片能替代的）

Step 10: 分析结果
  输入：实验数据（来自 Step 9）
  输出：LaTeX 分析段落
  卡片：实验结果分析

Step 11: 写论文
  输入：前面所有步骤的产出
  输出：完整的 LaTeX 论文
  卡片：Abstract + Introduction + Related Work + Method + Experiments

Step 12: 审稿自查
  输入：论文全文（来自 Step 11）
  输出：一致性问题 + 语言问题
  卡片：论文一致性审查 + 逐段语言审查

Step 13: 投稿
  输入：修改后的论文
  输出：格式检查 + 压缩方案
  卡片：投稿 Checklist + 论文压缩
```

## 四、目前卡片缺失的环节

对照上面的主线，我发现有几处断链：

### 缺失 1：Step 4「找迁移源」没有对应卡片

这是你直觉中最重要的一步。你看到一篇论文，第一反应是"这个方法能不能用到我的问题上"——这个判断目前没有卡片支持。

需要一张卡片叫类似 **「方法迁移评估」**：
- 输入：源论文的方法描述 + 我的目标问题描述
- 输出：迁移可行性分析（方法的核心假设是什么？在新场景下是否成立？需要做哪些适配？有哪些风险？）

### 缺失 2：从 Step 3 到 Step 4 的「跳转」不自然

目前「领域图谱梳理」输出的是「研究空白」，但这个空白是抽象的。用户需要的是更具体的指引："为了解决这个空白，你可以去 XX 领域看看"。

这可能需要增强「领域图谱梳理」的输出，让它除了指出空白，还推荐可能的迁移来源领域。

### 缺失 3：批量精读的「汇总」步骤

「论文精读笔记」是逐篇使用的，但从多篇笔记到「领域图谱梳理」之间，缺少一个「汇总整理」的步骤。用户需要手动把多篇笔记的内容拼在一起喂给「领域图谱梳理」。

### 缺失 4：实验执行阶段是「手动」的

Step 9（跑实验）完全靠用户自己。但 Step 8 到 Step 10 之间的衔接，可以有一张卡片帮助用户「规划实验执行顺序」——先跑什么、后跑什么、哪些可以并行。

## 五、流水线布局的设计思考

### 核心理念

用户从卡片池中选卡片，用箭头连接，形成一条自定义的「科研流水线」。箭头的意思是：上一张卡片的输出可以粘贴到下一张卡片的输入。

### 布局结构

```
                ┌─────────────┐
                │  卡片池（左侧） │
                │  ┌───┐ ┌───┐ │
                │  │ A │ │ B │ │
                │  └───┘ └───┘ │
                │  ┌───┐ ┌───┐ │
                │  │ C │ │ D │ │
                │  └───┘ └───┘ │
                └─────────────┘
                        │
                        │ 拖拽到画布
                        ▼
    ┌──────────────────────────────────────┐
    │           画布（右侧）                │
    │                                      │
    │   ┌─────────┐                        │
    │   │ 文献检索  │                        │
    │   │ 策略     │                        │
    │   └────┬────┘                        │
    │        │ (箭头：输出→输入)              │
    │        ▼                              │
    │   ┌─────────┐                        │
    │   │ 论文精读  │──┐                    │
    │   │ 笔记     │  │ (可多次使用)         │
    │   └─────────┘  │                    │
    │        ▲       │                    │
    │        └───────┘                    │
    │        │                             │
    │        ▼                              │
    │   ┌─────────┐                        │
    │   │ 领域图谱  │                        │
    │   │ 梳理     │                        │
    │   └────┬────┘                        │
    │        ▼                              │
    │   ┌─────────┐                        │
    │   │ Idea    │                        │
    │   │ 头脑风暴 │                        │
    │   └────┬────┘                        │
    │        │                             │
    │    ┌───┴───┐                         │
    │    ▼       ▼                         │
    │ ┌──────┐ ┌──────┐                   │
    │ │可行性 │ │创新点 │                   │
    │ │评估   │ │提炼   │                   │
    │ └───┬──┘ └───┬──┘                   │
    │     └────┬────┘                      │
    │          ▼                            │
    │    ┌──────────┐                      │
    │    │ 方法架构   │                      │
    │    │ 设计      │                      │
    │    └────┬─────┘                      │
    │         │                            │
    │    ┌────┴────┐                       │
    │    ▼         ▼                       │
    │ ┌──────┐ ┌──────┐                   │
    │ │损失函数│ │实验方案│                   │
    │ └───┬──┘ └───┬──┘                   │
    │     └────┬────┘                      │
    │          ▼                            │
    │    ... (后续步骤)                      │
    └──────────────────────────────────────┘
```

### 关键交互设计点

1. **箭头 = 数据流**：箭头不是"先做后做"的顺序关系，而是"A 的输出是 B 的输入"的数据关系。
2. **一张卡片可以被多个箭头指向**：比如「方法架构设计」的输出同时指向「损失函数」「实验方案」「Method 章节」。
3. **一张卡片可以出现多次**：比如「论文精读笔记」需要对多篇论文各用一次。可以设计成「循环节点」或者「实例化」的方式。
4. **预设模板 vs 自由搭建**：
   - 预设模板：提供几条常见的流水线（如"标准论文流程""精读+Idea流程""审稿准备流程"）
   - 自由搭建：用户完全自己选卡片、连箭头
5. **箭头上的「粘贴提示」**：hover 箭头时显示"A 的输出 XX 可以粘贴到 B 的输入 YY"

### 两种布局的关系

| | 默认布局（当前） | 流水线布局（新） |
|---|---|---|
| 用途 | 浏览所有卡片，按阶段分类 | 搭建具体的工作流，明确卡片间关系 |
| 组织方式 | 按 7 个阶段平铺 | 用户自定义的有向图 |
| 卡片间关系 | 隐含的（用户需要自己理解） | 显式的（箭头表示数据流） |
| 适合场景 | 刚开始用，了解有哪些卡片 | 已经确定要做一个具体项目 |

## 六、关于「A+B 是不是唯一范式」的回答

不是。但 A+B 是最可操作的。其他范式（新问题定义、理论分析等）需要的不是方法论，而是**洞察力**——这没法用卡片流程来实现。

不过有一件事卡片系统可以做：**降低「发现迁移源」的门槛**。

现在用户看到一篇论文，脑子里想的是"这个方法能不能用到我的问题上"，但这个判断全靠直觉。如果有一张卡片专门做这件事——输入源论文方法 + 目标问题，输出迁移可行性分析——那就能把这个模糊的直觉变成一个可操作的步骤。

这就是为什么我认为 Step 4（找迁移源）是主线中最关键的缺失环节。
